### 1. Yeachan-Heo/oh-my-codex — Codex CLI 的超级管理层（今日 ⭐+2,867，总 ⭐11,616）

> **仓库**：https://github.com/Yeachan-Heo/oh-my-codex
> **语言**：TypeScript | **协议**：MIT

**这是什么：** OmX（Oh My codeX）是一个 Codex CLI 的增强层，为 OpenAI Codex 添加了 hooks、Agent 团队管理、HUD 可视化界面等能力。名字致敬 Oh My Zsh，定位也类似——让 Codex 从"能用"到"好用"。

**核心功能：**
- **Hooks 系统**：在 Codex 执行前后注入自定义逻辑（pre/post-execution hooks）
- **Agent Teams**：并行管理多个 Codex Agent 实例，每个专注不同任务
- **HUD（Heads-Up Display）**：实时状态面板，可视化所有 Agent 的执行状态
- **可扩展架构**：插件式设计，允许社区贡献功能模块

**为什么今天爆火（+2,867⭐）：**
- OpenAI Codex 正成为 AI 编码的主流工具，But 原生体验粗糙
- OmX 解决了"多任务并行 + 执行可视化"的核心痛点
- TypeScript 实现，与 Node.js 生态天然融合

**后端开发者关注点：** Codex + OmX 的组合正在成为继 Claude Code 之后第二个高质量 AI 编码工作流。如果你在用 Codex CLI 做日常开发，OmX 的 hooks 系统可以与你的 CI/CD 流程深度集成。

### 4. iflytek/astron-rpa — 科大讯飞开源企业级 RPA 平台（今日 ⭐+141，总 ⭐7,316）

> **仓库**：https://github.com/iflytek/astron-rpa
> **官网**：https://agent.xfyun.cn
> **语言**：Java/Python/Node.js | **协议**：Apache 2.0

**这是什么：** 科大讯飞开源的企业级 RPA（Robotic Process Automation）桌面自动化平台，核心亮点是与 **Astron Agent**（https://github.com/iflytek/astron-agent）的深度双向集成，实现 Agent 智能决策 + RPA 自动化执行的闭环。

**技术架构：**
- **后端**：Java（JDK 8+）+ Python 3.13.x（RPA 引擎核心）+ Node.js 22+
- **包管理**：pnpm 9+ / UV 0.8+（Python）
- **部署**：Docker Compose + Casdoor（认证）
- **触发方式**：直接执行、定时任务、API 调用、**MCP 服务**

**300+ 预置原子能力：**
- Windows 桌面应用自动化（WPS、Office、金蝶、用友 ERP）
- 全系浏览器（IE/Edge/Chrome）自动化
- 数据处理、系统交互
- 企业内部系统（OA、ERP、CRM）

**与 Astron Agent 的集成：**
```
AI 推理（Astron Agent）→ 决策指令 → RPA 执行（AstronRPA）→ 结果回传 Agent
```
这种闭环设计让 Agent 不再只是"思考"，而是真正能控制桌面软件完成复杂业务流程。

**后端开发者关注点：** 如果你在做企业内部自动化流程，讯飞的 Astron 生态（Agent + RPA + MaaS）是目前国内最完整的一套。Apache 2.0 许可，无商业限制。

### 5. google/adk-java — Google Agent Development Kit Java 版（今日 ⭐+7，总 ⭐1,436）

> **仓库**：https://github.com/google/adk-java
> **Maven**：`com.google.adk:google-adk:1.0.0`
> **语言**：Java | **协议**：Apache 2.0

**这是什么：** Google 官方出品的 Java Agent 开发工具包，对标 Python 版 ADK（`google/adk-python`），让 Java 开发者可以用熟悉的语言构建生产级 AI Agent。

**核心特性：**
- **代码优先（Code-First）**：Agent 逻辑直接在 Java 中定义，而非 YAML 配置
- **丰富工具生态**：内置 GoogleSearchTool 等，支持 OpenAPI Spec 自定义工具
- **模块化多 Agent**：支持多 Agent 组合为层次结构
- **A2A 协议集成**：支持 Agent-to-Agent 远程通信（`google/A2A` 协议）
- **内置 Dev UI**：开发调试界面（与 Python 版一致）

**Maven 引入：**
```xml
<dependency>
  <groupId>com.google.adk</groupId>
  <artifactId>google-adk</artifactId>
  <version>1.0.0</version>
</dependency>
```

**Quick Start：**
```java
LlmAgent rootAgent = LlmAgent.builder()
    .name("search_assistant")
    .model("gemini-2.0-flash")
    .instruction("You are a helpful assistant.")
    .tools(new GoogleSearchTool())
    .build();
```

**Java 后端开发者关注点：** Spring Boot 项目接入 AI Agent 能力终于有了官方 Java SDK。结合 `spring-ai-community/spring-ai-agent-utils`（Claude Code-inspired tools for Spring AI），Java 生态的 Agent 开发工具链已相当完整。

### 6. DependencyTrack/dependency-track — 软件供应链安全分析平台（Java 安全重点）

> **仓库**：https://github.com/DependencyTrack/dependency-track
> **官网**：https://dependencytrack.org/
> **语言**：Java | **协议**：Apache 2.0 | **归属**：OWASP 官方项目

**这是什么：** OWASP 官方维护的软件供应链风险分析平台，通过 SBOM（Software Bill of Materials）追踪组织内所有应用的组件使用情况，识别已知漏洞、过期依赖、许可证风险。

**关键能力：**
- **SBOM 生产/消费**：标准 CycloneDX 格式（含 VEX 漏洞利用可交换性数据）
- **多源漏洞情报**：NVD + GitHub Advisories + Sonatype OSS Index + Snyk + Trivy + OSV
- **EPSS 集成**：Exploit Prediction Scoring System，优先处理最可能被利用的漏洞
- **CI/CD 原生**：API-First 设计，可集成到任何 CI/CD 管道
- **组件范围**：应用、库、框架、OS、容器、固件、硬件、服务

**后端开发者关注点：** 随着 Log4Shell 之后"软件供应链安全"进入监管视野，SBOM 已成合规要求。如果你的团队还没有系统化的依赖安全扫描，Dependency-Track 是 OWASP 认可的标准答案。结合 CycloneDX Maven Plugin 可自动生成 SBOM 并推送到平台。

## 三、🔍 趋势洞察与深度分析

### 趋势 1：AI 提示词工程的"安全边界"正在重新划定

`system_prompts_leaks` 持续 Trending 带来的最大冲击不是"泄露了什么"，而是**彻底改变了 AI 产品的安全假设**。

此前行业默认"系统提示词 = 安全边界"。现在这个假设已经失效。

**新的安全模型必须是：**
1. **代码层权限控制**：工具函数的执行权限由代码决定，而非提示词
2. **最小权限原则**：Agent 只能调用被明确授权的工具，无论提示词怎么说
3. **输出内容审计**：对 AI 输出进行独立安全扫描，不依赖自我约束

**给 Java 后端开发者的具体建议：** 在 Spring Boot + Spring AI 的 Agent 实现中，工具函数（`@Tool`）的实现里应该有显式的权限检查，不要把 "用户不能删数据" 写在提示词里——写在 Service 层的代码里。
