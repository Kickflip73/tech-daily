### 1. Poseidon-fan/linux-0.11-rs — Linux 0.11 用 Rust 重写（HN 81pt）

> **仓库**：https://github.com/Poseidon-fan/linux-0.11-rs
> **语言**：Rust | **协议**：MIT

**这是什么：** Linux 0.11（1991年 Linus Torvalds 最初版本）被完整用 Rust 重写，可以在 QEMU 中启动运行。

**技术意义：**
- 这是 Rust 作为系统编程语言的里程碑——证明 Rust 可以完全替代 C 实现操作系统内核
- Linux 0.11 约 1 万行 C 代码，重写后保持原有架构（8086 实模式 → 保护模式切换、进程调度、简单文件系统 ext2）
- 对于 Rust 后端开发者：理解内核级内存安全如何从语言层面消除 use-after-free、数据竞争等 C 经典 bug

**行业影响：** Rust 在操作系统领域的渗透加速。从 Redox 到 Linux 内核直接接受 Rust 驱动，再到今天的完整内核重写，Rust 正在蚕食 C 的最后堡垒。

### 7. xmz111/ReChannel — 从 RGB 生成到像素级密集预测（HF 9 upvotes）

> **论文**：arXiv:2607.06553 | **仓库**：https://github.com/xmz111/ReChannel

**这是什么：** 将预训练的文本-图像生成模型（DiT）适配为密集预测任务（深度图、法线图、分割等），**不额外生成 RGB 图像**，而是直接输出任务原生像素场。

**核心创新（ReChannel）：**
- 保留 DiT 的 VAE 编码器，丢弃目标侧解码器
- 用 **33K 参数**的 token-local 线性头，将每个 token 映射为其对应 p×p×K_t 像素空间
- 用 Task LoRA 微调冻结的 DiT

**性能数据：**
- 在 FLUX-Klein 4B 模型下，**超过所有现有 SOTA**（trimap-free 抠图、KITTI 深度、引用分割）
- 与"编辑+隐空间解码"方案相比：**准确率更高且 2.48x 更快**

**意义：** 生成模型的"RGB 输出接口"不再是唯一选择。密集感知任务可以直接从 DiT 的 token lattice 中读取，大幅减少参数和推理成本。

### 2. KronQ: LLM 量化 via Kronecker-Factored Hessian（HF 16 upvotes）

> **论文**：arXiv:2607.07964 | **仓库**：https://github.com/Intelligent-Computing-Lab-Panda/KronQ
> **机构**：University of Southern California

**核心问题：** GPTQ 等二阶 PTQ 方法只考虑输入激活统计，假设所有输出通道对重建目标贡献相等。

**KronQ 创新：**
1. **梯度协方差引入**：量化损失同时依赖激活协方差和梯度协方差
2. **双向非相干处理**：将输入侧随机旋转扩展到输出维度
3. **新敏感度指标**：基于梯度和激活 Hessian trace 的层间混合精度分配

**性能数据：** 2-bit 纯权重量化 LLaMA-3-70B → **仅 7.93 perplexity**（GPTQ/GPTAQ 在此配置下发散或产生退化量化 >2000 perplexity）

**后端开发者关注点：** 70B 模型 2-bit 量化后仍可实用，意味着推理成本可降低 4-8 倍。KronQ 是生产级部署的关键工具。

### 3. Scalable Visual Pretraining for Language Intelligence（HF 41 upvotes，最热论文）

> **论文**：arXiv:2607.09657

**颠覆性观点：** 传统 LLM 预训练假设"语言模型必须在纯文本上训练"，但大量知识通过图表、公式、页面布局传达。这些视觉线索在文本化过程中丢失。

**核心实验：** 对同一语料库，**视觉预训练（直接处理 PDF 图像）** 始终优于**纯文本预训练**，跨多个 backbone 和 benchmark。

**意义：**
- 挑战了 LLM 训练的基本假设
- 未来大模型可能直接从视觉文档学习，而非 OCR 后丢给文本模型
- 对 RAG 和知识库的影响：存储原始 PDF 图像 > 存储 OCR 提取文本

### 4. Shared Selective Persistent Memory for Agentic LLM Systems（HF 0 upvotes）

> **论文**：arXiv:2607.09493

**核心问题：** Agent 系统每次会话从零开始，丢失了之前会话积累的配置、模式、约束。

**解决方案（SSPM）：** 从完整历史中只保留 4 类可复用上下文：
1. 任务规范
2. 数据模式
3. 工具配置
4. 输出约束

**性能数据：**
- 任务完成率：**96%**（无记忆 79%，完整历史 71%）
- Zero-token 刷新：消除 LLM 重复调用，**14x 任务时间缩减**
- 摘要驱动生成：每轮 token 成本降低 **97x**

**关键发现：** 朴素地保存完整历史反而降低性能（71%），因为陈旧轨迹干扰决策。**选择性记忆** > 完整记忆 > 无记忆。

**后端开发者关注点：** Agent 系统的状态管理范式转变——不是"记住一切"，而是"记住该记住的"。

### 5. Failure as a Process: CLI Coding Agent Trajectory Analysis（HF 0 upvotes）

> **论文**：arXiv:2607.09510

**大规模研究：** 在 3,843 条执行轨迹（7 个前沿模型 × 3 个 Agent 框架）上分析 Agent 失败模式。

**14 个关键发现：**
- 失败主要源于**认知错误**（epistemic errors），而非技术错误
- 失败通常在**前几步**就开始，但隐藏到无法恢复时才显现
- 更早的验证和干预比最终结果评估更有效

**对 Agent 工程的启示：** 不能只看 Agent 最终是否成功，要监控执行轨迹的中间状态。Agent 越早失败，修复成本越低。
