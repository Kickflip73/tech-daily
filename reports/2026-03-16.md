### 1. volcengine/OpenViking — AI Agent 专属上下文数据库（今日 ⭐+1,870）

> **仓库**：https://github.com/volcengine/OpenViking
> **语言**：Python | **出品**：字节跳动火山引擎

**这是什么：** 字节开源的专为 AI Agent 设计的上下文数据库（Context Database）。传统 RAG 的痛点是：记忆散落在向量库、资源分布在各处、技能又在代码里——三者割裂，难以统一管理。OpenViking 用"文件系统范式"把三者合一。

**核心设计理念：**

**三层上下文结构（L0/L1/L2）：**
- **L0**：元信息层（目录结构、索引），每次都加载，极低 token 开销
- **L1**：摘要层（关键信息、近期上下文），按需加载
- **L2**：详情层（完整内容、原始数据），仅精确检索时拉取

**五大核心能力：**
1. **碎片化解决**：Memory（记忆）+ Resources（资源）+ Skills（技能）统一文件系统管理
2. **token 消耗降低**：L0/L1/L2 分层按需加载
3. **检索效果提升**：目录定位 + 语义搜索组合递归检索
4. **可观测上下文**：检索轨迹可视化，DEBUG 时清晰看到走了哪些目录
5. **Context 自演进**：自动压缩历史、自动归档任务记忆

**后端开发者关注点：**
- 如果你在做 Agent 平台或长期运行的 AI 任务流，这是目前设计最系统化的上下文管理方案
- L0/L1/L2 的分层思想与数据库的热温冷存储有异曲同工之妙，架构上值得借鉴
- OpenViking 的"文件即上下文"范式已被 OpenClaw（本系统所基于的平台）所采用——你现在用的 MEMORY.md 体系就是同类思想

### 5. p-e-w/heretic — LLM 自动去审查工具（今日 ⭐+1,062，总 ⭐14k）

> **仓库**：https://github.com/p-e-w/heretic
> **语言**：Python

**这是什么：** 一个自动移除开源 LLM 中"安全对齐"（拒绝回答）的工具。基于 Directional Ablation（方向消融）技术 + Optuna TPE 参数优化器，**全自动**找到最优消融参数。

**技术原理：**
- **Directional Ablation（消融术）**：在模型权重空间中找到表示"拒绝"方向的向量，然后在推理时从激活中减去该方向分量
- **优化目标**：同时最小化拒绝次数 + 最小化与原始模型的 KL 散度（保留智能）
- **全自动**：无需理解 Transformer 内部原理，命令行直接运行

**行业影响：** Heretic 持续高热度反映了 AI 安全对齐与开放使用之间的持续博弈。学术价值高（帮助研究对齐技术的脆弱性），但生产风险需评估。

### 2. Glassworm 卷土重来：Unicode 不可见字符攻击席卷 GitHub/npm/VSCode（205 points）

> **链接**：https://www.aikido.dev/blog/glassworm-returns-unicode-attack-github-npm-vscode

**这是什么：** 安全研究团队 Aikido 发现 Glassworm 攻击的新一波：攻击者在代码中注入不可见的 Unicode 特殊字符（零宽字符、方向控制符等），肉眼完全看不出来，但执行语义被篡改。

**攻击向量更新：**
- GitHub 代码审查界面不显示不可见 Unicode
- npm 包 README / package.json 被污染
- VSCode 某些主题/扩展渲染时隐藏危险字符

**防御建议：**
- `grep -rP '[\x00-\x08\x0b-\x0c\x0e-\x1f\x7f\xc2\xad\xe2\x80]'` 扫描仓库
- 在 CI 中加入 Unicode 安全扫描步骤
- 配置 VSCode 显示不可见字符：`"editor.renderControlCharacters": true`

**后端开发者关注点（高度相关）：** 供应链攻击 + AI 生成代码混合场景下，这类攻击更难被发现。Code Review 时务必开启字符可视化，对第三方依赖包保持警惕。

### 1. ExeVRM — 视频执行奖励模型，8B 模型超越 GPT-5.2 & Gemini-3 Pro

**技术亮点：**
- **ExeVR-53k 数据集**：53k 高质量 (视频, 任务, Reward) 三元组
- **对抗性指令翻译**：自动合成带步骤标注的负样本，提升模型判别能力
- **时空 token 剪枝**：移除同质化区域和静止 token，保留关键 UI 变化，解决长高清视频的 token 消耗问题
- **跨平台评估**：Ubuntu / macOS / Windows / Android 全覆盖

**性能（ExeVRM 8B）：**
- 准确率 84.7%、召回率 87.7%
- 超越 GPT-5.2 和 Gemini-3 Pro（两个超大模型！）

**意义：** 解决了 CUA（Computer-Use Agent）规模化评估的瓶颈问题，为 Agent 的 RLHF/强化学习提供可靠的自动奖励信号。

### 2. IndexCache — 跨层索引复用，稀疏注意力推理提速 1.82x

> **论文**：https://arxiv.org/abs/2603.12201
> **标题**：Accelerating Sparse Attention via Cross-Layer Index Reuse

**核心洞察：** DeepSeek Sparse Attention (DSA) 是生产级稀疏注意力方案，但其 Indexer（选择 top-k token 的组件）在每一层都独立运行，复杂度 O(L²)，开销不小。

**关键发现：** 相邻 Transformer 层的 top-k 索引选择高度相似（约 70-80% 重叠）。

**IndexCache 方案：**
- 将层划分为"Full 层"（运行自己的 Indexer）和"Shared 层"（复用最近 Full 层的 top-k 索引）
- **免训练版**：贪心搜索确定哪些层保留 Indexer，在校准集上最小化语言建模损失
- **训练感知版**：多层蒸馏损失，让每个 Indexer 服务其覆盖的所有层

**性能（30B 模型）：**
- 移除 75% 的 Indexer 计算，质量几乎无损
- Prefill 加速 **1.82x**，Decode 加速 **1.48x**

**后端开发者关注点：** 部署 DeepSeek 或类似 Sparse Attention 模型时，IndexCache 是低风险高收益的推理加速选项。开源实现预计随论文同步。

### 3. MADQA — Agent 到底是"战略导航"还是"随机试错"？

> **论文**：https://arxiv.org/abs/2603.12180
> **标题**：Strategic Navigation or Stochastic Search? How Agents and Humans Reason Over Document Collections

**核心问题：** 当前 Multimodal Agent 在文档密集型任务中，到底是在做真正的策略性推理，还是只是暴力搜索？

**方法：**
- **MADQA Benchmark**：2,250 个人工标注问题，基于 800 份异构 PDF 文档
- **评估协议**：accuracy-effort trade-off（准确率 vs 花费的查询次数）

**关键发现：**
- 最好的 Agent 在原始准确率上可以匹敌人类，但**成功的题目集合与人类几乎不重叠**
- Agent 依赖暴力搜索弥补弱战略规划——在无效路径上反复循环
- 与 Oracle 性能仍有接近 **20%** 的差距

**意义：** 正式量化了当前 Agent 的推理短板所在——不是"不够聪明"，而是"不够高效"。为下一步研究（强化规划能力）指明方向。

- **配套数据集**：稠密 3D 空间描述，引导模型以结构化方式更新快权重

**意义：** 自动驾驶、机器人、AR/VR 领域的核心能力——持续更新的空间记忆。在长程视频空间理解 benchmark 上达到 SOTA。

## 五、🔐 安全动态

### Glassworm Unicode 攻击防御清单

```bash
# 1. 扫描仓库中的不可见 Unicode
grep -rP '[\x00-\x08\x0b-\x0c\x0e-\x1f\u200b-\u200f\u202a-\u202e\ufeff]' ./src

# 2. Git pre-commit hook 检测
cat > .git/hooks/pre-commit << 'EOF'
#!/bin/bash
if git diff --cached --name-only | xargs grep -lP '[\u200b-\u200f\u202a-\u202e]' 2>/dev/null; then
    echo "⚠️  检测到不可见 Unicode 字符，提交已阻止"
    exit 1
fi
EOF
chmod +x .git/hooks/pre-commit

# 3. VSCode 配置显示不可见字符
# settings.json
{
    "editor.renderControlCharacters": true,
    "editor.renderWhitespace": "all"
}
```

**建议 Stark 先生将步骤 2 加入团队 CI 流程。** 在大量使用 AI 生成代码的当下，这类攻击被混入的风险大幅上升。
